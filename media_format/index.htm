Lifecyle Concerns for Archiving Digital Data

Documents in CHC Omeka applications are composed of descriptive metadata attached to one or more scanned pages or born-digital documents such as web pages. We will use the term Media to described the digital documents or pages associated with Omeka documents. Omeka provides a framework for managing, searching, discovering and viewing media and their metadata. More importantly, Omeka has to perform as a vehicle for taking the metadata and digital media into the future whence it will be necessary to systematically migrate the metadata and media to some unknown future management and publishing system.

There are many possible formats for media that can be created by the tools for scanning pages or saving born-digital documents. Not all of these possible formats are readable by the import tools in Omeka. In the long run we also have to be concerned about the media import ant export tolerances of the as-yet unknown tools that may be used to harvest and migrate the CHC-assets in the future. The best bet for assuring the future resilience of the digital collection is to avoid the use of any media formats that may cause problems in automated processing in the future.

Fortunately, digital archiving projects and interest groups have hashed and re-hashed the question of best practices for archiving digital media. We take guidance from the Digital Commonwealth

Quick Links for Scanning and Saving Archival PDFs

PDF format documents are easy to handle today. A hundred years from now, we probably should expect problems. Features that make PDFs compact today, such as relying on external font libraries or clever media or compression formats are all likely to cause problems in the future. Luckily for us, archivists have come up with a common denominator for packaging documents as Archival PDF documents. We can be reasonably sure that archival tools of the future will be able to read these and if necessary import these into the archival systems of the future, or to migrate these to digital document formats of the future, if necessary.

Tools For Handling and Checking Image Files

When scanning documents and manipulating the results, use a tool that allows you to

  • Inspect and set the color profile
  • Flatten the image to remove transparency
  • Set the compression format

To determine whether an image is formatted correctly, you can inspect its properties by right-clicking it in Windows, choose Properties, and then inspect the Details tab.

If you ware experimenting with a new workflow and want to test your images for compatibility with the Omeka importer, try importing the the image as media for an item in the Stage instance.

Recommendations for Image Files

The following properties can be evaluated for any image using your computer's file explorer. In Windows, right-click the image and choose properties.

TIFF Format

The TIFF format for digital images is a very flexible container for a varieety of image encodings some of which are considered solid for archival use and many others, that are less portable. While TIFF is recommended, it is necessary to look at specific properties of the TIFF file to assure that the image will be most likely to be preserved and transferrable for the long-term.

File Names:

Even though WIndows and Apple operating systems may be more flexible in terms of file and folder names, our folder and file names should be compatible with the more universal conventions used by Linux OS amd web applications.

  • Use only alphabetical and numerical characters plus "-", and "_".
  • No spaces or tabs in filenames!
  • Tif files should end with a simple ".tif" suffix.

Single-Page Media

Although it may be possible to save Tif files with multiple pages (or frames), these files cannot be loaded into Omeka and are also problematic in other archival management applications. Therefore all images should contain a single page or frame.

Color Representation / Bit Depth:

  • 8 bits per channel.
  • Gray Scale: One Channel -- 8 bits.
  • Color: 3 Channel RGB -- 24 bits.
  • Transparency: It is rare to see transparency in historical documents. When transparency is present in an image it is encoded as an additional channel (Alpha) which may result in a 4-channel, 32-bit image. This unnecessary complication causes problems occasionally. To remove transparency you should Flatten the image and check to make sure that the result has a bit-depth of 24. (8-bits per chanmel.)

Color Profiles:

Images exported from graphics and design applications often use color mapped images that apply Color Profiles other than simple RGB or sRGB. These should be avoided.

According to the NARA Guidelines (p38):

An ideal workflow would be to scan originals with a calibrated and characterized device, assign the profile of that device to the image file, and convert the file to the chosen workspace (Adobe RGB 1998 for color or Gray Gamma 2.2 for grayscale).

Compression:

  • No Compression
  • LZW Compression

Various scanners and file-save operations will default to other sorts of compression that will cause problems. Therefore it is important to choose tools and export functions that use No compression or LZW and make sure to check frequently that your images are safe.

Image Pixel Dimensions and DPI

Pixel Dimension os the count of pixels for the height and width of the rectangular image. In most cases, our digial documents are less than 30" x 40' Inches scanned at 200 dpi. This would be a maximum pixel dimension of 6000 x 8000 pixels. Or 11" x 17" at 300dpi or 3300 x 5500 pixels. Exceeding these tolerances is probably overkill for most images.
